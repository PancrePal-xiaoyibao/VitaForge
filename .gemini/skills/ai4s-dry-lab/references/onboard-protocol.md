# 追溯性入职协议 (Retroactive Onboarding Protocol)

> 适用场景：项目已进行到中途，ai4s-dry-lab skill 才被部署。
> 目标：扫描理解全部已有工作，反向填充 WORKLOG/STATE/SPEC/README，然后从当前研究边界开始正常 OODA 循环。

---

## 触发条件

以下任一情况触发本协议：
1. skill 首次激活时，检测到项目目录下已有脚本/输出，但缺少任务容器文档
2. 用户显式调用 `/ai4s-lab onboard`
3. skill 激活时 `docs/STATE.md` 不存在但 `scripts/` 或 `output/` 或 `results/` 目录非空

---

## 执行流程

### Phase 0: 项目结构扫描

```
1. 扫描顶层目录结构 (ls -la, find maxdepth=2)
2. 识别项目类型 (R/Python/混合，scRNA-seq/空间/其他)
3. 统计已有脚本数量、输出文件数量、数据文件数量
4. 输出扫描摘要
```

### Phase 1: 代码-输出溯源

对每个已存在的脚本文件，按时间/编号顺序逐一分析：

```
对 scripts/ 或项目根目录下每个分析脚本 (*.R, *.py):
  1. 读取脚本头部注释 → 提取用途描述
  2. 读取关键行 → 提取输入文件路径、输出文件路径、关键参数
  3. 检查输出文件是否实际存在
  4. 记录: {脚本名, 用途, 输入→输出映射, 关键参数}
```

可使用 Explore Agent 并行分析多个脚本以提高效率。

### Phase 2: 反向构建 SPEC

基于脚本分析结果，逆向推导：

```markdown
## 反向构建规则

1. 从脚本编号推断 Phase 划分 (如 7.1/7.2/7.3 → Phase 7)
2. 从脚本功能推断 SPEC 目标 (G1, G2, ...)
3. 从技术选型推断 ADR (如用 Slingshot 替代 Monocle3 → ADR)
4. 从参数设置推断常量表
5. 标记所有推断为 "[追溯推断]" 以区分后续新增内容
```

生成 `docs/SPEC-experiment.md`，头部标注：
```
> 本 SPEC 为 ai4s-dry-lab skill 追溯性构建，基于对已有代码的逆向分析。
> 推断内容标注为 [追溯推断]，需研究者确认。
```

### Phase 3: 反向填充 WORKLOG

对每个已识别的脚本/分析步骤，生成 WORKLOG 条目：

```markdown
### Step XXa: [脚本名] [追溯记录]
- **方法**: [从代码中提取的算法/工具/参数]
- **输入**: [从代码中提取的输入文件路径]
- **输出**: [从代码中提取或从文件系统验证的输出文件]
- **关键结果**: [从输出 CSV 中提取的关键数字]
- **代码位置**: [脚本路径]
- **SPEC 关联**: G{n} → ADR-{m}
- **OODA 状态**: Act (追溯记录默认标记为 Act)
- **文献验证**: "待验证"
- **记录类型**: [追溯填充 — AI 逆向分析生成，非实时记录]
```

**排序规则**: 按脚本编号或文件修改时间排序，最早的在最前。

### Phase 4: 生成输出目录 README

对每个包含分析结果的目录：

```
1. 列出目录内所有文件 (图形+数据)
2. 对每个图形文件:
   a. 找到生成它的脚本 (通过文件名关键词匹配脚本中的输出路径)
   b. 读取脚本中对应的生成代码段
   c. 从代码中提取: 数据源、计算逻辑、轴含义、颜色编码
   d. 生成解释文档
3. 对每个数据文件:
   a. 读取列名和前几行数据
   b. 找到生成脚本
   c. 生成列定义和计算逻辑说明
```

在 README 头部标注：
```
> 本文档为 ai4s-dry-lab skill 追溯性生成，解释内容源自对生成代码的逆向分析。
```

### Phase 5: 生成 STATE.md

基于以上分析结果，生成当前状态快照：

```markdown
## 当前位置
- Phase: [最后一个已完成步骤的 Phase]
- Step: [最后一个已完成步骤的编号]
- OODA: Act (追溯完成后标记为 Act)
- 最后操作: [最后分析的脚本的用途]
- 最后更新: [当前日期时间]

## 待办/阻塞
- [列出 WORKLOG 中标记为"未完成"或"失败"的项目]
- [列出缺失的分析 (SPEC 中有目标但无对应脚本)]
```

### Phase 6: 研究者确认

将以下内容呈现给研究者：
1. 追溯构建的 SPEC 摘要（目标列表 + Phase 划分）
2. WORKLOG 覆盖范围（哪些步骤已记录、哪些无法追溯）
3. STATE 当前位置
4. 未覆盖项列表（无脚本/无输出的目标）

等待研究者：
- 确认或修正 SPEC 目标和 Phase 划分
- 补充遗漏的信息
- 确认当前研究边界
- 指示下一步 OODA 方向

### Phase 7: 进入正常 OODA 循环

确认后，从 STATE 记录的当前 Phase + Step 开始正常的 OODA 循环。

---

## 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| 脚本无编号/无时间戳 | 按文件系统修改时间排序，标记为"[顺序待确认]" |
| 脚本有多个输出去向 | 全部记录，通过文件名关键词匹配 |
| 输出文件已被删除/移动 | 标记为"[输出文件缺失]"，从代码推断输出描述 |
| 脚本功能不明确 | 标记为"[功能待确认]"，请求研究者补充 |
| 脚本运行失败 | 从错误日志或代码分析推断意图，标记为"[失败]" |
| 脚本之间有数据依赖 | 通过输入输出路径交叉引用建立依赖关系图 |

---

## 效率建议

1. **并行分析**: 使用多个 Explore Agent 并行读取不同脚本
2. **分批处理**: 脚本数 > 10 时，按 Phase 分批处理
3. **增量补充**: 不追求一步到位，先建立骨架，后续 OODA 中逐步完善
4. **关键优先**: 优先分析有输出的脚本，跳过工具函数/配置文件
